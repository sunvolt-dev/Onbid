# 지번 기반 실거래가 정밀 매칭

## 배경

수익성분석 탭에서 국토부 실거래가를 조회할 때, 읍면동 + 건물명 텍스트 매칭 방식이 엉뚱한 거래를 매칭하거나 아예 데이터를 못 찾는 문제가 있다. 네이버부동산에서 주소로 검색하면 정확한 거래 내역이 나오는데, 현재 시스템은 그 수준에 미치지 못한다.

## 핵심 아이디어

국토부 API 응답의 `jibun`(지번) 필드를 새로 수집/저장하고, 온비드 물건의 지번과 매칭하는 Tier 0 단계를 추가한다. 지번은 필지의 고유 식별자이므로 같은 지번 = 같은 건물이 보장된다.

## 데이터 흐름

```
[온비드 물건]                         [국토부 API 응답]
onbid_cltr_nm / zadr_nm에서 추출:      XML item에서 추출:
  - 읍면동명                            - umdNm (읍면동)
  - 지번 (302-8)                        - jibun (302-8)  ← 신규 저장
  - 건물명 (숭의엠타운)                  - bldg_nm (건물명)

            ↓                                    ↓
       onbid_jibun                         cache_jibun
            ↓                                    ↓
            └──────── 매칭 ────────────────────────┘
```

## 매칭 알고리즘 (4단계 폴백)

| Tier | 조건 | 면적 제한 | 설명 |
|------|------|-----------|------|
| 0 (신규) | 같은 읍면동 + 같은 지번 | 없음 | 같은 건물 확정 |
| 1 (기존) | 같은 읍면동 + 같은 건물명 + 면적 ±30% | 있음 | 건물명 폴백 |
| 2 (기존) | 같은 읍면동 + 면적 ±30% | 있음 | 동 단위 비교 |
| 3 (기존) | 같은 시군구 + 면적 ±30% | 있음 | 시군구 단위 비교 |

Tier 0 매칭 시 해당 건물의 모든 거래를 보여줄 수 있어 면적이 다른 호수의 거래도 참고 가능.

## 지번 추출 로직

### 온비드 물건 (extract_jibun)

우선순위:
1. `zadr_nm`이 있으면 거기서 추출 (44/1875건만 해당)
2. 없으면 `onbid_cltr_nm`에서 추출 (대다수)

정규식: `r"(?:동|리|가)\s+(\d+(?:-\d+)?)"`
- 읍면동명 뒤에 오는 첫 번째 `숫자` 또는 `숫자-숫자` 패턴

입력 예시:
- `"인천 미추홀구 숭의동 302-8 외 4필지 숭의엠타운"` → `"302-8"`
- `"서울특별시 송파구 거여동 604-3 아피체"` → `"604-3"`
- `"울산 남구 신정동 1222-7 신정반트펠리시아"` → `"1222-7"`

### 국토부 API (XML 파싱)

`item.findtext("jibun")` — 모든 API 타입에서 제공되나 선택 필드(빈 값 가능).

### 지번 매칭 규칙

- 정확 매칭: `"302-8" == "302-8"`
- 본번 매칭: `"302-8"`의 본번 `"302"`와 `"302"` → 매칭 (부번 0은 생략되는 경우 있음)
- 빈 값: 어느 쪽이든 jibun이 없으면 Tier 0 스킵 → Tier 1로 폴백

## 변경 범위

### 1. schema_molit.py — MOLIT_TRADE_CACHE 스키마

`jibun TEXT` 컬럼 추가. 기존 데이터는 NULL로 유지, 30일 캐시 만료 시 재수집하면서 자동으로 채워짐.

### 2. molit_fetcher.py — XML 파싱

`_parse_xml_items()`에서 `jibun` 필드 추출 및 반환 dict에 추가.

### 3. molit_fetcher.py — 지번 추출 함수 (신규)

`extract_jibun(cltr_nm: str, zadr_nm: str | None) -> str | None`

### 4. molit_fetcher.py — 매칭 알고리즘

`match_trades()`에 `jibun` 파라미터 추가. Tier 0 매칭 로직 삽입.

### 5. molit_fetcher.py — get_market_price

`jibun` 파라미터 수신 및 `match_trades()`에 전달.

### 6. app.py — market-price 엔드포인트

`zadr_nm` 또는 `onbid_cltr_nm`에서 지번 추출하여 `get_market_price()`에 전달.

### 7. 프론트엔드 — TabProfit.tsx

변경 없음. API 응답 구조 동일, `match_tier` 값에 0 추가, `TIER_LABELS`에 Tier 0 라벨 추가만 필요.

## 기존 캐시 처리

- 기존 MOLIT_TRADE_CACHE 데이터의 jibun은 NULL
- ALTER TABLE로 컬럼 추가 (SQLite는 ADD COLUMN 지원)
- 즉시 반영 원하면 MOLIT_FETCH_LOG 삭제로 재수집 유도 가능
- 또는 30일 자동 만료까지 대기 (Tier 0 불가 시 기존 Tier 1~3 폴백)

## 정확도 우선 설계 원칙

- 속도보다 정확도 우선
- 지번 매칭 실패 시에만 기존 알고리즘으로 폴백
- 매칭 tier를 UI에 표시하여 사용자가 신뢰도 판단 가능
